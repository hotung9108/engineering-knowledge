# Logging

<details>
<summary>🇻🇳 <b>Hiển thị bản dịch Tiếng Việt</b></summary>
<br>

> **Tóm tắt**: Logging là việc chương trình của bạn "viết nhật ký" lại mọi thứ nó đang làm. Nó trả lời câu hỏi: **"Chuyện quái gì vừa xảy ra vậy?"**. Khi code chạy trên máy khách hàng và văng lỗi, bạn không thể cắm dây cáp vào máy tính của khách để debug. Bạn chỉ có thể dựa vào những dòng chữ mà Code đã chủ động viết ra file (Ví dụ: `Lúc 12:00, Khách hàng số 5 bấm nút thanh toán, lỗi vì thẻ hết tiền`). Nếu viết Log quá ít, bạn sẽ mù tịt không biết sửa lỗi. Nếu viết Log quá nhiều, bạn sẽ chìm trong rác và tốn hàng ngàn đô tiền lưu trữ ổ cứng. 

</details>

> **Summary**: Logging is the foundational pillar of Observability. It is the emission of immutable, timestamped, descriptive text records generated by the application code during execution. Logs answer the ultimate forensic question: **"What exact sequence of events occurred?"**. Because engineers cannot attach a debugger to a live production environment handling 10,000 concurrent requests, Logs provide the indispensable audit trail required to diagnose business logic failures and exceptions. Effective logging is a delicate balance: logging too little leaves you blind during an outage; logging too much exhausts disk I/O, clutters the aggregator, and incurs astronomical storage costs.

## ELI5 (Explain Like I'm 5)

<details>
<summary>🇻🇳 <b>Hiển thị bản dịch Tiếng Việt</b></summary>
<br>

Hãy tưởng tượng bạn thuê 1 ông Bảo vệ đứng canh cửa công ty.
1. **Bad Logging (Viết log dở)**: Ông bảo vệ ghi vào sổ: "Hôm nay có thằng nào đó đi vào, làm cái gì đó sai sai rồi đi ra". (Hoàn toàn vô dụng, khi sếp hỏi không biết đường nào mà tra cứu).
2. **Good Logging (Viết log xịn)**: Ông bảo vệ dùng biểu mẫu in sẵn, điền đúng cột: `[Ngày 15/10 - 08:30:15] [Cảnh báo Vàng] [Khu vực Kho] - Nhân viên Nguyễn Văn A (ID: 123) quẹt thẻ thất bại vì Thẻ hết hạn.`. (Cực kỳ rõ ràng, sếp nhìn vào biết ngay Lỗi do đâu, Ai làm, Lúc nào).

</details>

Imagine a Security Guard keeping a logbook at a bank vault.
1. **Unstructured/Poor Logging**: The guard scribbles: "Someone came in today and did something bad and it broke." (Completely useless. If a robbery occurs, the police have zero actionable data).
2. **Structured/Excellent Logging**: The guard uses a strictly formatted spreadsheet: `[TIMESTAMP: 2023-10-15 14:32:00] [SEVERITY: CRITICAL] [AREA: Vault 3] [ACTOR_ID: 994] [ACTION: Door Failed to Open] [REASON: Invalid Biometrics]`. (Perfect. When the system fails, engineers can instantly filter the spreadsheet and pinpoint the exact point of failure).

## Layer 1: What is it? (What)

<details>
<summary>🇻🇳 <b>Hiển thị bản dịch Tiếng Việt</b></summary>
<br>

Trong thế giới lập trình, có 5 cấp độ (Level) phổ biến để in Log, từ nhẹ đến nặng:
1. **DEBUG (Sửa lỗi)**: In ra mọi thứ. "Đang gán biến x = 5", "Đang mở cổng Database". Cực kỳ rác. CHỈ ĐƯỢC MỞ khi đang code ở máy cá nhân (Localhost).
2. **INFO (Thông tin)**: Các cột mốc quan trọng. "Hệ thống đã khởi động thành công", "Đã nhận đơn hàng số 100".
3. **WARN (Cảnh báo)**: Chuyện xấu vừa xảy ra nhưng chưa chết người. "User nhập sai pass 3 lần", "Database phản hồi hơi chậm (2 giây)". 
4. **ERROR (Lỗi)**: Có thứ gì đó vừa bị SẬP. "Không thể kết nối thanh toán Momo", "Bị đứt kết nối Database". (Bắt buộc phải lưu lại và báo động).
5. **FATAL (Tử vong)**: Lỗi tột cùng. Đứt mạng toàn hệ thống, tràn RAM, ổ cứng bốc cháy. Hệ thống tự động sập ngay sau khi in ra dòng Log này.

</details>

The industry standard strictly defines logging verbosity through 5 primary **Log Levels**:
1. **DEBUG (or TRACE)**: Extremely granular, hyper-verbose output used exclusively for step-by-step local development. (e.g., `Entering function CalculateTax() with args [A, B]`). **Rule**: Never enable in Production; it will immediately exhaust disk space.
2. **INFO**: Normal, significant lifecycle events confirming the system is operating correctly. (e.g., `Application Server started on Port 8080`, `Batch processing job completed`).
3. **WARN**: An unexpected or anomalous event occurred, but the system successfully recovered or mitigated it. The business flow continued. (e.g., `Primary DB timeout, successfully failed over to Secondary`, `Disk usage at 85%`).
4. **ERROR**: A specific operation or request catastrophically failed. Human intervention is likely required to fix the underlying bug. (e.g., `NullPointerException thrown during checkout`, `Payment Gateway rejected API key`).
5. **FATAL (or CRITICAL)**: An unrecoverable, systemic failure. The application is actively crashing or shutting down immediately after emitting this log. (e.g., `Cannot connect to Primary Database Cluster, halting application`).

## Layer 2: Why does it exist? (Why)

<details>
<summary>🇻🇳 <b>Hiển thị bản dịch Tiếng Việt</b></summary>
<br>

Khi code của bạn được đẩy lên Đám mây (Cloud), nó chạy bên trong một cái "Hộp đen".
Đột nhiên, khách hàng gọi lên tổng đài mắng: "Tại sao tôi bấm thanh toán, tài khoản ngân hàng bị trừ tiền, mà web lại báo lỗi?".
Bạn mở Code ra đọc. Code có hàng ngàn dòng, bạn chịu chết không thể đoán được nó bị kẹt ở dòng nào.
Lúc này, Log là chiếc phao cứu sinh duy nhất. Bạn mở hệ thống xem Log, gõ tìm số điện thoại của khách hàng đó. Màn hình hiện ra:
- `[INFO] Nhận request thanh toán từ SDT 090...`
- `[INFO] Đã trừ tiền thành công từ Ngân hàng`
- `[ERROR] Gửi Email xác nhận thất bại do mạng chậm quá 5 giây (Timeout)`
Nhờ có Log, bạn biết ngay thủ phạm là cái API Gửi Email bị chậm (chứ không phải code trừ tiền bị sai). Không có Log, Kỹ sư là những kẻ mù đường.

</details>

In local development, you debug by stepping through code using breakpoints. In a Cloud Native Production environment, the application is an opaque black box handling thousands of concurrent threads.
When a high-value customer complains: "My credit card was charged, but my account wasn't upgraded," you cannot reproduce their exact state locally. The state is ephemeral and gone.
Logs provide the definitive chronological audit trail of the system's internal state machine at the exact moment of failure. By querying the log aggregator (ELK/Datadog) for the user's `account_id`, the engineer instantly retrieves the story:
- `[INFO] Received upgrade request for account_id: 123`
- `[INFO] Stripe successfully charged $99`
- `[ERROR] TimeoutException connecting to legacy CRM Database. Aborting upgrade.`
Without logs, this diagnosis is physically impossible. Logs illuminate the black box.

## Layer 3: Without vs. With Comparison (Compare)

<summary>🇻🇳 <b>Hiển thị bản dịch Tiếng Việt</b></summary>
<br>
So sánh cách viết Log của một Lập trình viên mới ra trường và một Senior.

Visualizing the difference between Unstructured and Structured Logging.

| Metric | Unstructured (Junior Dev) | Structured JSON (Senior Dev) |
|---|---|---|
| **Code** | `log.info("User " + id + " paid " + amt);` | `log.info("payment_success", {userId: id, amount: amt});` |
| **Output File** | `User 99 paid $50` | `{"event":"payment", "userId":99, "amount":50, "status":"ok"}` |
| **Machine Readability**| Horrible. Requires complex Regex to parse. | Perfect. Databases natively index JSON keys. |
| **Searchability** | Cannot easily filter by users who paid > $100. | Query: `SELECT * WHERE amount > 100` works instantly. |

## Layer 4: Common Use Cases

<details>
<summary>🇻🇳 <b>Hiển thị bản dịch Tiếng Việt</b></summary>
<br>

1. **Gom Log tập trung (Centralized Logging - ELK)**: Bạn có 50 cái máy chủ nhỏ (Microservices). Bạn không thể mở 50 cái màn hình đen thui lên để đọc Log từng cái được. Người ta cài một phần mềm chuyên đi "hút" Log từ 50 cái máy đó, ném vào 1 cái kho dữ liệu trung tâm (Elasticsearch). Sau đó bạn chỉ cần mở 1 trang web duy nhất (Kibana), gõ từ khóa, nó sẽ tìm kiếm trên toàn bộ 50 máy đó trong 1 giây.
2. **Cảnh báo tự động (Log-based Alerts)**: Nếu file Log có xuất hiện chữ `FATAL` hoặc `ERROR` xuất hiện 100 lần trong 1 phút. Hệ thống gom Log sẽ tự động hú còi, nhắn tin SMS gọi Kỹ sư dậy chữa cháy.
3. **Phân tích Hành vi (Audit / Analytics)**: Log không chỉ để bắt Lỗi. Nếu bạn in log mọi hành động `[INFO] User X vừa bấm vào nút Y`. Bạn có thể đưa đống Log này cho team Phân tích Dữ liệu (Data Team) đếm xem hôm nay nút Y có bao nhiêu người bấm.

</details>

1. **Centralized Aggregation (ELK / Fluentd / Datadog)**: Microservices are ephemeral. When a Kubernetes Pod crashes, it is physically destroyed, and its local log files are permanently deleted. Centralized Logging solves this by running a sidecar daemon (Fluentd/Logstash) that continuously tails the local `stdout` streams and aggressively ships the logs over the network to a permanent, persistent, centralized search engine (Elasticsearch or Splunk) *before* the pod dies.
2. **Log-Based Alerting**: Logs are strictly reactive forensic tools, but their aggregators are proactive. You configure rules in Datadog: `IF count(level:ERROR AND service:payment) > 50 WITHIN 5_minutes THEN trigger PagerDuty`. The logging pipeline acts as the system's immune system, automatically paging engineers when abnormal patterns emerge in the text streams.
3. **Security Auditing & Compliance (SIEM)**: Highly regulated industries (Banking, Healthcare) are legally mandated (PCI-DSS, HIPAA) to log specific user actions. Logs detailing `Who accessed which record at what timestamp` are streamed to WORM (Write Once Read Many) storage. Security teams use tools like Splunk SIEM to parse these logs, detecting malicious intrusion attempts or insider data theft based on abnormal log patterns.

## Layer 5: Deep Practice

### Best Practices

<details>
<summary>🇻🇳 <b>Hiển thị bản dịch Tiếng Việt</b></summary>
<br>

1. **BẮT BUỘC dùng Structured Logging (Log theo chuẩn JSON)**: Quên ngay cái việc dùng `console.log("Câu văn")` đi. Máy tính (Datadog, ElasticSearch) rất ghét đọc câu văn. Hãy dùng các thư viện như `Winston` (NodeJS) hoặc `Logback` (Java) để in Log ra dưới định dạng JSON. Bạn sẽ lọc được Log theo các cột như `user_id` hay `ip_address` một cách thần tốc.
2. **Kẹp Trace ID vào mọi dòng Log**: Nếu bạn đang sửa lỗi cho 1 Request đi qua 5 Service. Nếu Log của 5 Service đó không có ID chung, bạn không thể ghép nối câu chuyện lại được. Ngay ở cửa vào (Gateway), hãy đẻ ra 1 cái mã `Trace_ID = ABC`. Sau đó cứ truyền cái mã `ABC` này đi khắp mọi nơi. Mọi dòng Log in ra phải có kèm `Trace_ID: ABC`. Lúc sửa lỗi, search 1 phát chữ `ABC` là lòi ra toàn bộ đường đi.

</details>

1. **Strictly Enforce Structured Logging**: The era of parsing flat text files with `awk` and `grep` regex is over. Application logs MUST be emitted as highly structured JSON payloads. Instead of formatting strings, provide explicit Key-Value context. Example: `log.error("Payment failed", {"user_id": user.id, "transaction_id": tx.id, "error_code": 500})`. Modern aggregators (Elasticsearch) instantly parse this JSON, creating strongly-typed indexes for `user_id` and `transaction_id`, enabling hyper-efficient multidimensional filtering.
2. **Contextual Injection (Trace IDs & MDC)**: In a highly concurrent environment, 500 threads are writing to the same log file simultaneously. The lines interleave chaotically. To untangle the chaos, you must use Mapped Diagnostic Context (MDC in Java) or AsyncLocalStorage (Node.js). You inject a `Trace_ID` at the Edge Gateway. The framework automatically appends this `Trace_ID` to *every single log statement* executed by that specific thread. Searching Datadog for that `Trace_ID` perfectly filters the chaos into a clean, linear, single-request timeline.

### Common Pitfalls

<details>
<summary>🇻🇳 <b>Hiển thị bản dịch Tiếng Việt</b></summary>
<br>

1. **Lộ lọt Dữ liệu Nhạy cảm (PII / Tội ác bảo mật)**: Bạn in `[INFO] Đã tạo User thành công: { email: "a@a.com", password: "123", the_tin_dung: "4111-..." }`. Dòng Log này được ném lên Datadog. Toàn bộ Lập trình viên trong công ty đều có thể lên Datadog và đọc được Mật khẩu, Thẻ tín dụng của khách hàng. Công ty bạn sẽ bị kiện sấp mặt và đi tù.
   - *Luật*: BẮT BUỘC phải viết 1 hàm `Masking` (Che mặt) can thiệp vào bộ in Log. Cứ thấy cái nào là `password` hay `credit_card` thì tự động đổi thành `***` trước khi in ra ổ cứng.
2. **Nuốt Exception (Swallowing Errors)**: Junior dev thường có thói quen viết khối `try/catch`. Trong hàm `catch(Exception e)`, chỉ in ra đúng câu `log.error("Lỗi rồi!")` rồi cho chương trình chạy tiếp. Hành động này ĐÃ XÓA SẠCH DẤU VẾT CỦA BỘ NHỚ (Stack Trace). Mất Stack Trace là mất đi dòng số mấy, file nào gây ra lỗi. 
   - *Luật*: Phải in toàn bộ cục Exception ra: `log.error("Lỗi thanh toán", e)`.

</details>

1. **Leaking PII (Personally Identifiable Information) & Secrets**: The most devastating compliance violation. A developer innocently logs an entire incoming HTTP Request payload to debug a parsing issue. That payload contains unhashed Passwords, Social Security Numbers, and Credit Card PANs. These secrets are now broadcasted in plaintext to Splunk/Datadog, accessible to dozens of engineers, violating PCI-DSS and GDPR. **The Fix**: The Logging Framework must be configured with Data Scrubbing/Masking interceptors. Before any JSON is serialized to `stdout`, a regex engine must proactively mask (e.g., `****`) keys matching `password`, `ssn`, or `token`.
2. **Swallowing Stack Traces**: A catastrophic developer anti-pattern. Writing `catch (Exception e) { log.error("DB Query Failed: " + e.getMessage()); }`. This prints "DB Query Failed: Null pointer", effectively truncating and permanently destroying the deeply nested Stack Trace. The engineer is robbed of the exact filename, class, and line number that triggered the fault. **The Fix**: Always pass the Raw Exception Object directly to the logger (e.g., `log.error("DB Query Failed", e)`). The framework will automatically unroll and print the full, multiline Stack Trace.
